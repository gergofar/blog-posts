A demo that works once, on a hand-picked input, in front of a friendly
audience, is a fundamentally different artifact from a system real users
depend on. The gap between the two is where most AI initiatives quietly
stall.

![From a single working demo to a resilient production system](https://cdn.jsdelivr.net/gh/gergofar/blog-posts@main/images/demo-vs-production.svg)

## The demo is the easy 20%

Getting a model to do something impressive once is genuinely easier than it
has ever been. The hard part starts the moment you ask: *what happens on the
input you didn't plan for?*

- **Reliability** — does it degrade gracefully, or confidently produce
  nonsense?
- **Evaluation** — can you measure whether a change made it better or worse,
  before your users do?
- **Operability** — who owns it at 2am when it breaks?

| Dimension | Demo                   | Production                 |
| --------- | ---------------------- | -------------------------- |
| Inputs    | Curated, happy path    | Adversarial, long tail     |
| Failure   | Acceptable, off-camera | Paged, with an owner       |
| Quality   | "Looks great!"         | Measured, regression-gated |

> A demo answers *can it work?* A product answers *can I depend on it?* — and
> those are different engineering problems.

## Evaluation is the unlock

If I could give one piece of advice to a team moving from prototype to
product, it would be: **build the evaluation harness before you scale the
model.** Treat outputs like any other thing you'd test.

```python title="evals.py" {3}
def regression_check(model, cases):
    failures = [c for c in cases if not passes(model(c.input), c.expected)]
    return len(failures) / len(cases)  # the number that gates every release
```

It's not glamorous. But a team that can measure quality can improve it
deliberately — and that's the difference between a demo and a product.
